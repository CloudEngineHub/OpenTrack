# tracking 部署使用说明（2026-05-13）

## 1. 文档范围

这份 README 只解释这套 tracking 部署代码怎么用，不重复 SDK 分层和头文件索引。

它重点回答 3 个问题：

1. 这套程序怎么编译、怎么启动。
2. `start_deploy.sh` 启动后，遥控器按键如何驱动状态机和 tracking policy 切换。
3. 想替换新的 policy 或新的参考动作时，文件应该放到哪里，代码应该改哪里。

---

## 2. 一句话理解这套 tracking 工程

这套程序的运行路径可以直接理解为：

1. `./build.sh` 编译出 `build/bin/state_machine_example`。
2. `./start_deploy.sh` 启动状态机程序。
3. 程序启动时会把 `robot_controller.hpp` 里配置好的 tracking controller 全部构造出来，并预加载对应的 `policy.onnx`。
4. 控制线程真正开始工作后，用遥控器把机器人切到 `Stand`、`Loco`，再通过方向键组合切到某个 tracking controller。
5. tracking controller 一边读取真机状态，一边读取 `storage/data/<data_name>/ref_data.onnx` 中的参考动作，再调用 `storage/policy/<policy_name>/.../policy.onnx` 生成目标关节角。

所以这套工程的核心不是改 CMake，而是维护两类运行时资源：

1. `storage/policy/` 下的策略网络。
2. `storage/data/` 下的参考动作数据。

## 3. 最小启动流程

### 3.1 编译

在 `G1_deploy` 目录下执行：

```bash
./build.sh
```

脚本会清理旧的 `build/`，重新执行 CMake，并产出：

```text
build/bin/state_machine_example
```

### 3.2 启动

继续在 `G1_deploy` 目录下执行：

```bash
./start_deploy.sh
```

这个脚本会：

1. 进入 `build/bin/`。
2. 运行 `./state_machine_example eth0 --param ...`。
3. 在 `build/bin/logs/<时间戳>/` 下创建日志目录。

有两个使用层面的注意点：

1. 启动早期会有一段模型预加载时间，因为 `RobotController` 构造函数里把所有已配置的 tracking controller 都创建了一遍。
2. tracking policy 本身并不从 `--param` 指向的目录读取模型，而是固定从 `storage/policy/` 和 `storage/data/` 取文件。
3. 当前 `main.cpp` 只解析了 `--param`，脚本里传进去的第一个网卡参数并没有真正传给 `InitDdsModel()`；如果后续要显式绑定网卡，不能只改脚本，还要补主程序参数解析。

### 3.3 推荐的上机顺序

推荐按下面的顺序操作：

1. 运行 `./start_deploy.sh`。
2. 等终端持续提示 `Press R2 to start!`。
3. 按一次 `R2`，启动控制线程。
4. 按一次 `A`，进入站立状态。
5. 按一次 `X`，进入 locomotion 状态。
6. 如果要切换策略族，按 `B` 轮换模式。
7. 在 `LOCO` 状态下，按方向键或组合键，切到某个 tracking policy。

如果你跳过第 4、5 步，直接按方向键，大部分 tracking 切换不会生效，因为 tracking 状态只能从 `LOCO` 进入。

## 4. 启动后的状态机逻辑

### 4.1 状态枚举

当前状态机只有 4 个状态：

1. `DAMPING`
2. `STAND`
3. `DANCE`
4. `LOCO`

这里的 `DANCE` 在这套项目里可以直接理解为“某个 tracking controller 正在执行”。

### 4.2 基本按键

| 按键 | 作用 | 说明 |
| --- | --- | --- |
| `R2` 单击 | 启动控制线程 | 只是开始控制循环，不代表切到 tracking |
| `A` 单击 | 切到 `STAND` | 允许从 `DAMPING`、tracking、`LOCO` 回站立 |
| `X` 单击 | 切到 `LOCO` | 允许从 `STAND` 或 tracking 回 locomotion |
| `B` 单击 | 切换模式组 | 在 4 组 policy 映射之间轮换 |
| `F2` | 切到 `DAMPING` | 代码实际是 `F2` 单键触发，不是注释里写的 `L2 + B` |
| `F1` | 退出程序 | 触发 `stop_requested_` |
| `Y` | 播放音频测试 | 与 tracking 本身无关 |

### 4.3 `B` 键切换的 4 个模式组

`B` 每按一次，`mode_index` 在下面 4 组里轮换：

1. `0`：普通模式
2. `1`：抗扰动模式
3. `2`：地形模式
4. `3`：基线模式

也就是说，方向键组合对应的“槽位编号”不变，但槽位里装的是哪一个 policy，会随着 `mode_index` 改变。

### 4.4 方向键与 tracking 槽位

tracking 的按键映射是一个 `4 x 20` 的表，定义在 `robot_controller.hpp` 里的 `controller_mapping`。

20 个槽位按下面的规则编号：

1. `0` 到 `3`：单独按 `上 / 下 / 左 / 右`
2. `4` 到 `7`：`L1 + 上 / 下 / 左 / 右`
3. `8` 到 `11`：`R1 + 上 / 下 / 左 / 右`
4. `12` 到 `15`：`L2 + 上 / 下 / 左 / 右`
5. `16` 到 `19`：`R2 + 上 / 下 / 左 / 右`

当前工程里这 20 个槽位大致按动作族组织：

1. 单独方向键：舞蹈类
2. `L1 + 方向键`：对抗类
3. `R1 + 方向键`：跳跃和跑步类
4. `L2 + 方向键`：起身/摔倒类，当前有不少槽位还是空的
5. `R2 + 左右`：额外槽位，当前放了一些 real clip 或 terrain 相关策略

需要注意两个运行时行为：

1. 某些槽位在某些模式下是 `nullptr`，按下去不会切换。
2. tracking 状态不能直接切到另一个 tracking 状态，因为 `toDance()` 只允许从 `LOCO` 进入。想换下一个动作，要么等当前 tracking 自然结束自动回到 `LOCO`，要么先按 `X` 回 `LOCO`，再重新选择。

### 4.5 tracking 结束后的行为

`FsmTrackerController` 跑到参考动作末尾后，会把 `dance_done_flag` 置为 `true`。控制线程检测到这个标志后会自动：

1. 把状态改回 `LOCO`
2. 把当前控制器切回 `FsmLocoController`
3. 调用 `Reset()` 重新进入走路策略

所以 tracking 播完后，不会停在原地等待，而是自动回到 walking policy。

## 5. tracking policy 的文件组织

当前 tracking controller 的构造函数需要两个字符串：

1. `policy_name`
2. `data_name`

它们分别决定两条路径。

### 5.1 policy 路径

```text
storage/policy/<policy_name>/checkpoints/
```

运行时会做两件事：

1. 读取 `storage/policy/<policy_name>/checkpoints/config.json`
2. 扫描 `checkpoints/` 下所有“纯数字命名”的子目录，自动选取数值最大的那个目录，并加载其中的 `policy.onnx`

也就是说，实际生效的模型路径是：

```text
storage/policy/<policy_name>/checkpoints/<最大数字目录>/policy.onnx
```

### 5.2 参考动作路径

```text
storage/data/<data_name>/ref_data.onnx
```

这个 `ref_data.onnx` 不是策略网络，而是参考轨迹数据，当前 controller 会从里面读取：

1. `qpos`
2. `qvel`
3. `feet_height`
4. `root_height`

## 6. 如何替换一个已有槽位的 policy

最常见的需求其实有 3 种，处理方式不一样。

### 6.1 只换策略网络，不换参考动作

这种情况最简单。比如你想保留 `dance1_subject1` 的参考轨迹，只把策略换成新训练结果。

操作步骤：

1. 把新策略放到 `storage/policy/<new_policy_name>/checkpoints/`。
2. 保证 `checkpoints/config.json` 存在，并且里面有 `env_config.obs_keys`。
3. 在 `checkpoints/` 下放好至少一个数字目录，例如 `004000317440/`，目录内包含 `policy.onnx`。
4. 修改 `state_machine/robot_controller.hpp` 对应槽位的构造参数，把原来的 `policy_name` 改成新的名字。
5. `data_name` 保持不变。
6. 重新编译并重启程序。

例如原来是：

```cpp
new FsmTrackerController("09060825_G1Tracking_ours_dance1_subject1_p1e-4_w1e-4", "dance1_subject1")
```

如果只换 policy，可以改成：

```cpp
new FsmTrackerController("my_new_tracking_policy", "dance1_subject1")
```

### 6.2 同时换策略网络和参考动作

如果新策略对应的是一段全新的动作，那么 `policy_name` 和 `data_name` 一般都要换。

操作步骤：

1. 把策略放到：`storage/policy/<new_policy_name>/checkpoints/.../policy.onnx`
2. 把参考动作放到：`storage/data/<new_data_name>/ref_data.onnx`
3. 修改 `state_machine/robot_controller.hpp` 对应槽位：

```cpp
new FsmTrackerController("<new_policy_name>", "<new_data_name>")
```

4. 重新编译并重启程序。

### 6.3 只想让同一个 policy 目录切到更新的 checkpoint

如果你不想改代码，只是想让已有 `policy_name` 自动加载更晚的训练结果，那么可以直接：

1. 在同一个 `storage/policy/<policy_name>/checkpoints/` 下新增一个更大的数字目录。
2. 把新的 `policy.onnx` 放进去。
3. 重启程序。

因为当前代码会自动选择“最大数字目录”，所以不需要改 `robot_controller.hpp`。

## 7. 如何准备新的 `ref_data.onnx`

如果你拿到的是轨迹原始数据，而不是现成的 `ref_data.onnx`，可以参考仓库根目录的 `npz2onnx.py`。

它的作用是把参考轨迹整理成一个常量 ONNX 模型，并导出：

1. `qpos`
2. `qvel`
3. `feet_height`
4. `root_height`

实际使用时，你只需要保证最终产物落到：

```text
G1_deploy/storage/data/<data_name>/ref_data.onnx
```

至于中间是不是用 `npz2onnx.py` 生成，并不影响部署侧代码。

---

## 8. 如果要改“哪个按键触发哪个动作”

如果你不是想替换模型文件，而是想改“按哪个键触发哪个动作”，核心修改点只有一个：

```text
state_machine/robot_controller.hpp
```

具体做法是修改 `RobotController` 构造函数里的 `controller_mapping[mode_index][slot]` 初始化表。

你可以把它理解为两层映射：

1. 第一层：`mode_index` 决定当前处于普通 / 抗扰动 / 地形 / 基线 哪一组。
2. 第二层：`slot` 决定当前按键组合对应哪个 `FsmTrackerController(policy_name, data_name)`。

如果只是想把某个动作从 `L1 + up` 换到 `R1 + left`，本质上就是把对应的 controller 指针从一个槽位挪到另一个槽位。

## 9. 运行时几个容易混淆的点

最后列几个最容易踩到的点：

1. `R2` 在程序启动前后有两层含义：启动阶段按一次 `R2` 是“开始控制线程”；进入控制线程后，`R2 + 方向键` 又可以作为 tracking 槽位组合键。
2. tracking 不是热更新的。只要你改了 `storage/policy/` 或 `storage/data/` 的内容，最稳妥的做法都是重启进程。
3. `X` 对应的是 walking policy，不是 tracking policy。它加载的是单独的 `storage/policy/G1-Walk.onnx`。
4. 当前 `start_deploy.sh` 里有网络接口和 `--param` 参数，但 tracking 模型装载的真正关键路径还是 `storage/policy/` 和 `storage/data/`。
5. 如果你按了某个组合键没有反应，优先检查 3 件事：
   1. 现在是不是已经在 `LOCO` 状态。
   2. 这个槽位在当前 `mode_index` 下是不是 `nullptr`。
   3. 对应目录下的 `policy.onnx`、`config.json`、`ref_data.onnx` 是否都存在。

以上这些弄清楚之后，这套 tracking codebase 的日常使用基本就够了。
